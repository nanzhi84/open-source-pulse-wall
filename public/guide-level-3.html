4. 命令行路径

命令行不是为了显得专业，而是为了让学生理解 Git 的真实动作。前两章已经让学生完成了贡献，这一章再把每条命令拆开，学生会更容易把命令和结果对应起来。

完整命令

git clone https://github.com/your-github-id/open-source-pulse-wall.git
cd open-source-pulse-wall
git remote add upstream https://github.com/nanzhi84/open-source-pulse-wall.git
git checkout -b add-your-profile
cp data/profiles/_template.json data/profiles/your-github-id.json
npm run validate
git add data/profiles/your-github-id.json
git commit -m "feat: add my profile"
git push origin add-your-profile

逐条解释

`git clone`

把自己的 Fork 下载到本地。终端当前在哪个目录，仓库就会被下载到哪个目录下面。

`cd open-source-pulse-wall`

进入项目根目录。后面的 `npm run validate` 和 Git 操作都应该在这里执行。

`git remote add upstream`

给老师仓库起一个远程名字。以后同步老师仓库更新时会用到它。

`git checkout -b`

创建并切换到任务分支。这个分支只放本次 Profile 修改。

`cp`

复制模板文件。Windows PowerShell 里也可以用 `Copy-Item`，但课堂可以统一用 VS Code 文件树复制。

`npm run validate`

运行项目自带校验脚本。它比肉眼检查可靠，能发现字段和 JSON 格式问题。

`git add`

选择要进入 commit 的文件。这里明确写文件路径，是为了避免把无关文件一起提交。

`git commit`

保存一次变更快照。提交信息应该短，但要说清楚目的。

`git push`

把本地分支推到自己的 GitHub Fork。推送后才能在网页上创建 PR。

推荐在每一步后检查

git status
git branch
git remote -v

`git status` 用来看工作区有没有未提交文件；`git branch` 用来看当前在哪个分支；`git remote -v` 用来看 `origin` 和 `upstream` 是否指向正确仓库。新手不要等到最后一步才检查，因为路径或远程仓库一旦弄错，后面每一步都会跟着错。

常见失败

  • `npm run validate` 提示找不到脚本：通常是终端没有进入项目根目录。
  • `git push` 要求登录：需要完成 GitHub 认证，可以改用 VS Code 的 Source Control 推送。
  • PR 里出现很多文件：说明 `git add` 范围太大，提交前没有看 `git status` 或 diff。
  • 分支名写错：不影响内容，但会让 PR 不好读。分支名建议能说明任务。
