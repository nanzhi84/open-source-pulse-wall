3. VS Code 全流程

这一章解决网页端流程的两个短板：学生容易找不到仓库目录，也不容易提前发现 JSON 格式错误。VS Code 把文件树、编辑器、终端和 Git 面板放在同一个窗口里，适合作为新手的主路径。

准备工作

  • 电脑已经安装 VS Code。
  • 电脑已经安装 Git，VS Code Source Control 面板能识别 Git 仓库。
  • 电脑已经安装 Node.js 18 或更高版本，用来运行 `npm run validate`。
  • GitHub 上已经 Fork 了老师仓库。

克隆仓库

  1. 打开 VS Code，进入 Source Control 面板，选择 Clone Repository。
  2. 粘贴自己 Fork 的仓库地址。地址应该包含自己的 GitHub 用户名。
  3. 选择一个固定目录，例如桌面上的课堂文件夹。不要随手选下载目录，也不要选到系统目录。
  4. 克隆完成后，VS Code 会询问是否打开仓库。选择打开。
  5. 打开后看左侧文件树，应该能看到 `data`、`public`、`scripts`、`test` 等目录。

添加 Profile 文件

  1. 在文件树里打开 `data/profiles`。
  2. 复制 `_template.json`，把新文件命名为 `your-github-id.json`。
  3. 打开新文件，只修改自己的内容，不改 `_template.json`。
  4. 如果 VS Code 出现红色波浪线，先停下来修 JSON 格式。常见原因是漏双引号、漏逗号、最后一项多逗号。
  5. 保存文件后，再进入终端校验。

运行校验

打开 VS Code 内置终端时，要先确认终端路径在项目根目录。如果终端里看不到 `package.json` 所在目录，`npm run validate` 可能会报找不到脚本。

npm run validate

校验通过，说明 JSON 能被项目脚本读取，字段也符合当前规则。校验失败时，先读错误里出现的文件名和字段名，不要盲目重写整个文件。

提交和推送

  1. 打开 Source Control 面板，先看 Changed Files。
  2. 正常情况下，只应该出现你自己的 `data/profiles/your-github-id.json`。
  3. 点击文件查看 diff。绿色是新增内容，红色是删除内容。
  4. 确认无误后 stage 文件，提交信息写 `feat: add my profile`。
  5. 点击 Sync Changes 或 Push，把分支推到自己的 GitHub Fork。
  6. 回到 GitHub 页面，创建 Pull Request。

新手检查点

  • 窗口标题或文件树根目录显示的是 `open-source-pulse-wall`。
  • 终端能执行 `npm run validate`，而不是提示找不到 `package.json`。
  • Source Control 里没有出现 `public`、`dist`、`README.md` 等无关文件。
  • 提交前已经看过 diff，确认没有改到别人的 Profile。
