2. GitHub 网页端贡献

这一章适合完全没有本地环境的学生。目标不是把 Git 学完，而是先完成一次真实贡献闭环：在自己的 Fork 里改文件，然后用 Pull Request 把修改交回老师仓库。

适用场景

网页端流程适合第一次上课、没有安装 Node.js、没有配置 VS Code、或者只是想快速体验 PR 的学生。它的优点是路径短，所有操作都在浏览器里完成；缺点是本地校验能力弱，不适合长期开发。因此这一章只负责建立信心，后面还要进入 VS Code 和命令行。

操作步骤

  1. 打开老师仓库，点击页面右上角的 Fork，把仓库复制到自己的 GitHub 账号下。
  2. 进入自己的 Fork，确认页面左上角显示的是自己的用户名，而不是老师的用户名。
  3. 打开 `data/profiles` 目录，先点开 `_template.json` 看结构，不要直接修改模板。
  4. 新建文件 `data/profiles/your-github-id.json`，把 `your-github-id` 换成自己的 GitHub 用户名。
  5. 复制模板内容，填写 `name`、`github`、`role`、`motto`、`stack`、`city`、`style`、`avatar`、`homepage` 和 `cheers`。
  6. 提交更改时，选择创建新分支。分支名可以写 `add-your-profile` 或 `add-你的用户名-profile`。
  7. 提交后点击 Compare & pull request，确认 base 是老师仓库的 `main`，compare 是你自己的分支。
  8. 填写 PR 标题，例如 `feat: add my profile`，然后创建 Pull Request。

网页端能完成贡献，但不容易提前发现 JSON 格式错误。只要进入 VS Code 或命令行路径，就必须运行 `npm run validate`。

提交前检查

  • 你是否在自己的 Fork 里操作，而不是老师仓库里操作。
  • 文件是否放在 `data/profiles` 目录下，而不是仓库根目录。
  • 文件名是否是自己的 GitHub 用户名，例如 `octocat.json`。
  • `github` 字段是否和文件名一致。
  • `cheers` 是否保持为空数组 `[]`。
  • Pull Request 的目标是否是老师仓库的 `main`。

常见误区

很多学生会把网页端的“提交文件”理解成已经完成开源贡献。实际上，提交只发生在自己的 Fork 或自己的分支里。只有创建 Pull Request，老师仓库才会收到这次修改。还有一种常见错误是把 base 和 compare 选反，导致 PR 看起来像是要把老师仓库合并进自己的 Fork。创建 PR 时要停下来检查一次方向。
